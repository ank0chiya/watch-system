# エージェント 動作確認ガイド

本ドキュメントは、コンテナ化された監視エージェント（`agent`）が正常に起動し、ホストマシンのリソース情報を収集し、Consul経由で見つけたAPIサーバーへデータを送信できているかを確認するための手順である。

---

## 1. コンテナの起動状態確認

エージェントが正しくデプロイされ、プロセスが安定しているかを確認する。

### 1-1. コンテナステータスの確認
1. ターミナルで以下を実行：
   `docker compose ps agent`
2. **期待結果:** `STATUS` が `Up` または `running` になっていること。

### 1-2. 起動ログとメトリクス収集の確認
1. ターミナルで以下を実行：
   `docker compose logs -f agent`
2. **期待結果:** 以下のログが順に出力されていること。
   - `エージェントを起動しました [Host: (ホストID), Interval: 5s]`
   - `送信成功: CPU: ...%, Memory: ...% -> http://apiserver:8080/api/v1/metrics`
   - ※ `CPUUsage` や `MemoryUsage` が 0.0% ではなく、ホストの実際の負荷を反映した数値になっていることを確認する。

---

## 2. サービスディスカバリとレジリエンスの確認

分散システムとしての「賢さ（自己修復能力）」を確認する。

### 2-1. 起動直後の待機挙動（ヘルスチェック待ち）
1. 全サービスを同時に起動（`docker compose up -d`）した直後のログを確認する。
2. **期待結果:**
   - 起動直後に `警告: APIサーバーが見つかりません` が数回出ても正常。
   - 数秒後、Consulのヘルスチェックがパスしたタイミングで自動的に `送信成功` に切り替わること。

### 2-2. サービス追従テスト（APIサーバー停止）
1. APIサーバーを一時停止させる：
   `docker compose stop apiserver`
2. エージェントのログを確認する。
   - **期待結果:** `警告: 健全なAPIサーバーインスタンスが見つかりません` と表示され、エージェントがクラッシュせずにリトライを続けること。
3. APIサーバーを再開させる：
   `docker compose start apiserver`
4. **期待結果:** 数秒以内に自動で `送信成功` が再開されること。

---

## 3. エンドツーエンド（E2E）のデータ到達確認

エージェントが収集したデータが、最終地点である Redis のキューまで到達しているかを確認する。

### 3-1. Redis Streams の確認
1. Redis Master コンテナ内でストリームの末尾を確認する：
   `docker exec -it redis-master redis-cli xrange metrics_stream - +`
2. **期待結果:**
   - エージェントのログに表示されていた CPU/メモリの数値と一致するデータが保存されていること。
   - 送信元ホストの ID が正しく記録されていること。

---

## 4. トラブルシューティング

もし動作が期待通りでない場合は、以下の「大本」を確認する。

- **`no such host` エラーが出る場合:**
  - 実行時のコンテナ間通信（DNS）に失敗している。全サービスが `networks: - watch-net` に所属しているか、`daemon.json` の設定が干渉していないかを確認する。
- **`健全なインスタンスが見つかりません` が続く場合:**
  - APIサーバー側でエラーが起きている。`docker compose logs apiserver` を確認し、Consulへの登録が成功しているかチェックする。
- **CPU/メモリが常に 0.0% になる場合:**
  - `docker-compose.yml` で `pid: "host"` が設定されているか確認する。これが無いとコンテナ自身の（極めて低い）負荷しか計測できない。
